继承-Inheritance

  • 使用关键字extends

  • java中的类仅仅支持单继承[避免出现网状结构,保证类的层次性],但是java中的接口是支持多继承的.

    一个子类只能拥有一个直接的父类. 一个父类可以拥有多个子类.

  • 如果一个类没有明确表示它继承的父类,那么这个类将会默认继承java.lang.Object

  • 子类可以共享/访问/使用 父类中所有的非私有的成员.

  • 什么地方允许使用继承? - 不能为了代码的复用而乱用继承 - 满足 B is A , B extends A

  • 把多个类的共性抽离出来,放在父类中.子类中是允许出现个性的成员.

  • 父类的构造方法是不能够被子类继承

  • 子类是拓展父类的功能

优点

提供了代码的复用性.方便程序的拓展

符合软件开发设计原则 - 开闭原则 - 对内禁止修改,对外允许拓展

RootV1 - 功能1 ,功能2

RootV2 extends RootV1 功能3

缺点

  1. 修改了父类,可能会影响子类功能的使用

  2. 继承加强了类与类之间的耦合性 - 违背了软件开发设计的原则 - 高内聚,低耦合

    高内聚 - 单个模块独立完成业务的能力比较强

    低耦合 - 模块之间的关系比较弱

重写特点

解答题 - 重写和重载的区别

  1. 发生的前提 - 必须要有继承

  2. 重构 - 高度保持一致[修饰符,返回类型,方法名,方法的参数列表,方法抛出的异常列表]

  3. 重写的方法的名称以及参数列表必须要一致.

  4. 修饰符 - 子类的修饰符访问权限不能低于父类的访问权限

    访问修饰符由高到低public>protected>默认的>private

  5. 子类重写的方法的返回类型可以和父类的方法的返回类型一致,也可以是父类方法返回类型的子类.

  6. 子类方法抛出的异常列表和父类方法抛出的异常列表一致,或者是父类方法抛出异常类的子类.

子类一旦重写了父类的方法,运行的时候,就会执行子类重写之后的方法.

不可修改的.

  1. final修饰局部变量[方法体中的变量,包括形参]

  2. final修饰 属性[全局变量] - 常量属性 - 不可变

    通常static结合一起使用

  3. final修饰的类 - 不可被继承

  4. final修饰的方法 - 不可被重写

多态 - Polymorphism

定义:同一个行为具有多个不同表现形式或形态的能力

同一个动作作用于不同的对象,会有不同的解释,产生不同的执行结果.

多态的应用 - 面向父类编程/面向接口编程

  1. 对象的编译时类型写成父类,对象的运行时类型写成子类

  2. 方法的返回类型写成父类,方法的执行结果写成子类

  3. 方法的参数类型写成父类,可以传入这个父类的任何一个子类对象

static修饰的成员初始化的时机是在jvm加载类进内存的时候,并且机会只有一次,它在内存中的实例只有1个.

  2. static修饰的成员[属性/方法] - 属于类拥有的 - 静态的属性变量 - 类变量,并且是对象共享的.

  3. static作为静态导入

  4. 静态内部类

  5. 静态代码块

jvm加载类进内存的活动的顺序

  1. 对所有的静态的属性分配空间以及初始化

  2. 对所有的静态的方法分配空间以及初始化

  3. 调用静态代码块

  4. 如果遇到创建对象

    4-1. 给所有的非静态的属性分配空间以及初始化

抽象类

  1. 使用abstract修饰的类代表是一个抽象类,抽象类本身是不能够被实例化的.

  2. 抽象类天生就是用来被继承的

  3. 只有抽象类/接口才有资格去定义抽象方法(abstract修饰的方法)

  4. 一个普通类如果继承了一个抽象父类.那么这个普通类必须要重写这个抽象父类中提供的所有的抽象方法.

    除非这个普通类本身也是一个抽象子类.

  5. 抽象类中允许存在构造

  6. 抽象类中允许普通属性和普通方法

  7. 抽象父类也是允许拥有多个子类的.

不能一起使用的修饰符

  1. private 和 abstract

  2. static 和 abstract

    static修饰的方法不具备多态 - 并且不能够被重写.

    abstract就是为了多态,就是为了被子类重写

  3. final 和 abstract

    final修饰的类不能被继承

    final修饰的方法不能被重写.

缺点:产生大量的工厂实现类.不太好维护.

设计原则

  1. 开闭原则 - 对外允许拓展,对内禁止修改.

  2. 单一职责原则 - 一个类/接口不要完成多个业务.不要让一个类/接口太”重量”

  3. 依赖倒置原则 - 由维护具体到维护抽象

  4. 接口隔离原则 - 如果某个实现类没有必要实现该接口中的所有的抽象方法.有必要拆分接口.

  5. 迪米特原则 - 最少知道原则. 一个类直接”知道”另外一个类,越少越好.

  6. 里氏替换原则 - 能够用父类的地方,必然使用子类.建议子类不要去重写父类的方法.子类不要随意

    去破坏父类定义好的结构.

  7. 合成复用原则 - 由于继承带来代码的侵入性.鼓励使用合成复用原则来替代继承的写法.

装饰器模式

装饰器模式（Decorator Pattern）允许向一个现有的对象添加新的功能，同时又不改变其结构。这种类型的设计模式属于结构型模式，它是作为现有的类的一个包装。

这种模式创建了一个装饰类，用来包装原有的类，并在保持类方法签名完整性的前提下，提供了额外的功能。

重写clone方法

需要被克隆的对象需要使用java.lang.Cloneable标记接口,否则调用clone方法的时候抛出java.lang.CloneNotSupportedException

  1. 浅克隆 - 浅拷贝 - 浅层复制 - Object中的默认的clone就是浅克隆,依然是需要重写的.

    因为object中使用protected修饰

    特点

    • 基本类型/String/Date - 修改不会影响到克隆对象的属性值

java.math.Decimal

提供执行任意精度整数运算（ BigInteger ）和任意精度十进制运算（ BigDecimal ）的类。

构造

  1. BigDecimal(String val)
  2. BigDecimal(double val)

常用方法

  1. BigDecimal add(BigDecimal augend)
    返回 BigDecimal ，其值是 (this + augend) ，其标为 max(this.scale(), augend.scale()) 。

  2. BigDecimal divide(BigDecimal divisor)
    返回BigDecimal ，其值为(this / divisor) ，优先级为(this.scale() - divisor.scale()) ; 如果不能表示确切的商（因为它具有非终止的十进制扩展），则抛出一个ArithmeticException 。

  3. BigDecimal subtract(BigDecimal subtrahend)
    返回 BigDecimal ，其值是 (this - subtrahend) ，其标为 max(this.scale(), subtrahend.scale()) 。

  4. BigDecimal multiply(BigDecimal multiplicand)
    返回 BigDecimal ，其值是 (this × multiplicand)，其标为 (this.scale() + multiplicand.scale()) 。

枚举类型

在jdk5.x开始使用枚举类型来替代常量接口的使用 - 管理整个应用中的常量.

提供可读性

  1. 关键字enum来定义枚举类型,编译之后也是一个枚举常量
  2. 自定义的枚举类型默认都是会继承java.lang.Enum<E extends Enum> 抽象类
  3. 枚举类型中可以提供普通属性和普通方法
  4. 不能有public,protected构造,只能使用private以及默认
  5. 枚举类型不能够在外部new
  6. 每个枚举常量都是代表这个枚举类型的唯一实例
  7. 允许提供抽象方法,但是枚举常量必须要实现这个抽象方法.
